SAFE WORKFLOW

Plan → Build → Review → Test

PLAN

先理解，不修改

請 agent 列出現況、科學假設、風險、修改檔案與驗證方法。

BUILD

一次只做一個小步驟

建立函式、schema 或 tests，避免一次生成整個系統。

REVIEW

使用唯讀 reviewer

檢查資料洩漏、單位、時間窗、偏差與過度宣稱。

TEST

讓結果可重現

執行 tests、資料重建與 diff 檢查，最後才 commit／push。

高品質 Prompt 模板

閱讀 AGENTS.md 與相關檔案。

目標：完成一個單一、可驗證的任務。
輸入：列出資料路徑、schema、時間範圍與單位。
限制：不修改 data/raw；不捏造資料、結果或引用；先規劃再修改；不 push。
輸出：指定程式、CSV、JSON、圖或 Markdown 報告。
驗證：tests、邊界案例、指標與人工檢查。

完成後回報：
1. 修改檔案
2. 執行命令
3. 測試結果
4. 科學假設
5. 未解決限制

REAL EXAMPLES

四個地震研究實際案例

01

處理地震資料

檢查 catalog schema、時間、經緯度、深度、規模、缺值、重複事件，再審查 MAXC 與移動窗。

@seismic-data
閱讀 data/sample/catalog.csv 與 completeness.py。
先完成 QC，不修改原始資料。
檢查 MAXC binning、tie handling、3 個月移動窗與低樣本標記。
完成後執行 tests。
02

撰寫科學論文

先建立 evidence matrix，再由 writer 撰寫，最後用唯讀 reviewer 挑戰論述。

@manuscript-writer
只根據 src、configs、tests 與 outputs 撰寫 Methods。
不得補寫不存在的方法；未驗證引用標成 [AUTHOR CHECK]。
03

建置監測系統

把 Raspberry Pi 系統拆成資料接收、buffer、QC、震度、輸出、監控與斷線恢復。

使用 Plan agent 規劃 Raspberry Pi 地震監測 MVP：
連續波形、ring buffer、每秒震度、gap handling、
斷線恢復、健康監測與中央端 JSON。
先規劃，不寫程式。
04

互動網頁與 GitHub Pages

Python 產生科學結果 JSON，前端只負責視覺化，再用 Actions 測試與發布。

/build-dashboard
新增逐秒震度時間序列頁面。
保持手機友善；不在瀏覽器重新計算科學結果；
完成後執行 tests 與靜態檔案檢查。

AUTOMATION

用 CLI 建立可重複的研究工作

批次輸出 QC JSON

mkdir -p outputs
opencode run \
  --agent seismic-data \
  -f data/sample/catalog.csv \
  --format json \
  "只做資料品質檢查，不修改檔案" \
  > outputs/catalog-qc.jsonl

共用 Headless server

# Terminal A
opencode serve --port 4096

# Terminal B
opencode run --attach http://localhost:4096 \
  "檢查 tests 是否覆蓋 Mc 邊界條件"
